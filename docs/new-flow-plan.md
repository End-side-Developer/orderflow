## Product Context For Future AI Reasoning

OrderFlow is a Theme 11 government workflow app. Its product claim is: turn court judgments into verified, owner-assigned, deadline-safe actions.

The new target flow is not a single upload-time AI generation. It is a gated five-stage wizard:

1. Page extraction after the user clicks Intake.
2. Full judgment summary after page extraction is done and reviewed.
3. Action plan generation only after the user clicks Generate Action Plan.
4. Human review where each item can be approved, edited, rejected, or regenerated individually.
5. Trusted dashboard that shows only human-approved action records.

The trust boundary is human verification. The app must always communicate that AI output is decision support, not final legal advice. Every important AI claim must link to source page and excerpt. Low-confidence records must be marked for human review.

The cost boundary is caching. AI should run once per valid cache key and be reused on refresh or reopening the same PDF. Regeneration must be surgical: one page, one action item, one timeline, or one directive, never the whole document unless the user explicitly asks.

Architecture:

- `app/frontend`: Next.js App Router and reviewer UI.
- `app/backend`: FastAPI API, persistence, extraction, proof, routing, audit, and tests.
- `app/worker`: Temporal worker for durable workflow orchestration.
- `app/intelligence`: LangGraph/intelligence assets and deterministic tests.
- `app/infra`: Docker Compose stack for Postgres, Redis, MinIO, Temporal, OTel, LibreTranslate.
- `app/shared`: shared contracts and schema artifacts.

- Persisted documents, clauses, obligations, workflow runs, page summaries, page annotations, case flow, advocates, and places.
- PDF parsing with Docling-preferred path and deterministic fallback.
- Obligation extraction with confidence annotations and citation spans.
- Reviewer decisions via `PATCH /api/v1/obligations/{obligation_id}`.
- Workflow status routes under `/api/v1/workflows`.
- Page summary API and persistence under `page_summaries`.
- Case flow graph component and case incidence map component.
- Frontend upload workspace, obligations board, dashboard, document summary flow page, PDF viewer, and AI overlays.

Important current gaps:

- `app/worker/src/orderflow_worker/workflows/intake.py` is still a stub workflow.
- `app/worker/src/orderflow_worker/activities/intake.py` contains stub activities.
- `page_summaries` lacks `content_hash`, `prompt_version`, `source_excerpt`, and `ai_token_usage`.
- `obligations` lacks action-plan lifecycle fields such as `nature_of_action`, `action_plan_stage`, `regen_count`, and `regen_history`.
- There is no `document_summaries` cache table yet.
- There is no per-case gated wizard at `/case/[id]` yet.

## Non-Negotiable Design Decisions

- Restructure the existing pipeline; do not rebuild a second app beside it.
- Extend existing `page_summaries` and `obligations` records unless a task explicitly says otherwise.
- Keep old routes as compatibility shims until the new wizard is working.
- The worker owns long-running orchestration. The frontend polls or listens; it does not run workflow logic.
- The backend owns gate checks and returns `409` for invalid stage transitions.
- The dashboard API must filter to approved records only.
- Cache validity must include document identity, page number when relevant, content hash when relevant, prompt version, and model/provider when relevant.
- Prompt/model version changes are valid reasons to regenerate.
- Rate-limit handling must wait, reduce concurrency, and resume from the failed page without losing previous page results.
- Map flow is conditional. Do not force a map when places are weak or clustered.

## Agent Role Boundaries

- Backend AI: FastAPI routes, schemas, migrations, persistence, extraction, tests.
- Worker AI: Temporal workflows, activities, retry/resume behavior, worker tests.
- UI AI: Next.js routes, components, hooks, API client, UI tests.
- QA AI: targeted tests, manual verification scripts, quality gate notes.
- Codex/integrator: cross-service alignment, plan updates, final wiring, conflict resolution.

Any agent may read across services, but should only write inside the ownership needed for its one selected task.
